# CLAUDE.md

## What is this

t3code-api is a REST-to-WebSocket bridge for T3 Code. It connects to a running T3 Code server over WebSocket, buffers orchestration events in memory, and exposes a simple HTTP API for external applications.

This is a standalone sidecar — it does NOT modify T3 Code. It's just another WebSocket client that happens to re-expose the functionality over REST.

## Tech stack

- **Runtime**: Bun
- **HTTP framework**: Hono (minimal, Bun-native)
- **WebSocket**: Native `WebSocket` (Bun built-in)
- **Dependencies**: Only `hono` — intentionally minimal

## Project structure

```
src/
├── index.ts          # Entry point, config from env vars, HTTP server setup
├── ws-client.ts      # Persistent WebSocket client to T3 Code (auto-reconnect, request/response matching)
├── event-buffer.ts   # Per-thread circular event buffer (orchestration.domainEvent)
├── routes.ts         # REST endpoints (Hono app) — translates HTTP ↔ WS protocol
└── openapi.yaml      # OpenAPI 3.1 spec — served at /openapi.yaml, powers Swagger UI at /docs
```

## How it works

1. On startup, opens a WebSocket to the T3 Code server (`T3API_WS_URL`)
2. Listens for `orchestration.domainEvent` push messages and buffers them per thread
3. Incoming HTTP requests are either:
   - **Reads** from the event buffer (messages, events, status) — served immediately
   - **Commands** forwarded to T3 Code via WebSocket (create thread, send message, interrupt) — awaits WS response

## Environment variables

| Variable | Default | Purpose |
|----------|---------|---------|
| `T3API_PORT` | `4774` | Bridge HTTP port |
| `T3API_WS_URL` | `ws://localhost:3773` | T3 Code WebSocket endpoint |
| `T3API_TOKEN` | _(none)_ | Bearer auth token for the bridge |
| `T3API_MAX_EVENTS` | `500` | Max events per thread buffer |

## Key design decisions

- **Polling over SSE/streaming**: External consumers poll for events. Simpler to implement and consume from any language.
- **Event buffer is ephemeral**: In-memory only, lost on restart. T3 Code server is the source of truth — use `GET /snapshot` or `orchestration.replayEvents` to rebuild.
- **No schema validation on the bridge side**: Requests are passed through to T3 Code which validates them. The bridge is a thin proxy, not a gatekeeper.
- **IDs generated server-side**: `threadId`, `messageId`, `commandId` are generated by the bridge (UUID v4) so the caller doesn't need to.

## T3 Code WebSocket protocol (reference)

The bridge speaks the same protocol as the T3 Code web frontend. Key shapes:

**Request** (bridge → T3 Code):
```json
{ "id": "1", "body": { "_tag": "method.name", ...params } }
```

**Response** (T3 Code → bridge):
```json
{ "id": "1", "result": { ... } }
```

**Push** (T3 Code → bridge, unsolicited):
```json
{ "type": "push", "sequence": 42, "channel": "orchestration.domainEvent", "data": { ... } }
```

Protocol definitions live in T3 Code at `packages/contracts/src/ws.ts` and `packages/contracts/src/orchestration.ts`.

## API docs

Swagger UI is served at `/docs`, the raw OpenAPI spec at `/openapi.yaml`. Both are exempt from bearer auth so they're always accessible. The spec is a static YAML file (`src/openapi.yaml`) loaded at request time — no code generation involved.

## Adding new endpoints

1. Add the route in `src/routes.ts`
2. For read operations: use `events.getEvents()` or add a new method to `EventBuffer`
3. For write operations: use `ws.request("method.tag", { ...params })` to forward to T3 Code
4. Update `src/openapi.yaml` with the new endpoint schema
5. The T3 Code WS method tags are documented in `packages/contracts/src/ws.ts` (`WS_METHODS` and `ORCHESTRATION_WS_METHODS` constants)
